# Canon character models — identity manifest

These GLBs were generated by Tripo3D and uploaded with **Claude-assigned filenames that were
partly wrong** (Tripo names files generically; earlier passes mislabeled several Maime meshes as
"BANNON_*"). Identity here is established from the **baked texture atlas** (the only ground truth —
material UUID is stable across every duplicate export) plus the owner's tells:

> BANNON = steel mask + free-floating jaw plate + blond/brown dreadlocks + vale tudo trunks.
> MAIME  = white skull/clown facepaint + short twisted hair + jeans-with-chains / tattered gear.

## Rig state — why some are "static"
Every Tripo GLB ships **zero skinned bones**. The *rigready* exports split the body into 15 mesh
nodes named exactly `pelvis/chest/head/shL/shR/elL/elR/haL/haR/hipL/hipR/knL/knR/ftL/ftR`. The
engine loader (`_bindFighterGltf`) now detects those names, builds a humanoid skeleton at `REST`,
and attaches each part to its like-named joint so the existing retarget drives it (auto-fires; no
`window.GORO_RIG` flag needed). Files marked **single-mesh** are one un-split mesh — they load but
stay a static prop until a *rigready* (named-part) export of the same mesh is provided by the Tripo
pipeline. Same fix, they just need the split.

## Wiring
Registered in `MODEL_LIBRARY` (BANNON_v150.html) with `charKey` so they appear as selectable
alternate attires per fighter. These are the **Three.js-era** attires; per the roadmap the AAA target
is the C++ port, where these same identities carry over as authored bodies.
